# 数字人情感陪伴系统 - 从0到1实施规划

> 本文档提供完整的、可执行的项目实施方案，适合毕业设计使用

## 一、项目技术栈

### 1.1 前端技术
```
- 框架：Vue 3 (Composition API)
- 构建工具：Vite
- UI组件库：Ant Design Vue
- 状态管理：Pinia
- 路由：Vue Router
- HTTP客户端：Axios
- 语音：Web Speech API
```

### 1.2 后端技术
```
- 框架：FastAPI (Python)
- 数据库：MongoDB
- AI模型：qwen3-omni-flash (通过API)
- CORS：FastAPI CORS中间件
- 认证：简单的Session/Token
```

### 1.3 开发工具
```
- Node.js：v18+
- Python：3.9+
- MongoDB：6.0+
- 代码编辑器：VS Code
- API测试：Postman（可选）
```

## 三、MongoDB数据库设计

### 3.1 数据库配置
```javascript
数据库名：digital_companion
连接方式：使用MongoDB MCP服务器
```

### 3.2 集合设计

#### Collection 1: users
```javascript
{
  _id: ObjectId,
  username: String,              // 用户名
  session_id: String,            // 会话ID
  created_at: Date,              // 创建时间
  last_active: Date,             // 最后活跃时间
  preferences: {
    theme: String,               // 主题
    voice_enabled: Boolean       // 是否启用语音
  }
}
```

#### Collection 2: digital_persons
```javascript
{
  _id: ObjectId,
  user_id: ObjectId,             // 关联用户
  name: String,                  // 名称
  relationship: String,          // 关系（奶奶/朋友/导师）
  personality: String,           // 性格（温柔/幽默/智慧）
  avatar: String,                // Base64头像
  custom_prompt: String,         // 自定义提示词
  created_at: Date,
  updated_at: Date
}
```

#### Collection 3: conversations
```javascript
{
  _id: ObjectId,
  user_id: ObjectId,
  digital_person_id: ObjectId,
  messages: [
    {
      role: String,              // 'user' 或 'assistant'
      content: String,           // 消息内容
      timestamp: Date,
      emotion: String            // 情感标签
    }
  ],
  context_summary: String,       // 上下文摘要
  created_at: Date,
  updated_at: Date
}
```

#### Collection 4: emotional_analysis
```javascript
{
  _id: ObjectId,
  user_id: ObjectId,
  conversation_id: ObjectId,
  emotion_trend: [
    {
      timestamp: Date,
      emotion: String,           // 'positive', 'neutral', 'negative'
      score: Number              // 0-1
    }
  ],
  key_topics: [String],
  created_at: Date
}
```

#### Collection 5: career_suggestions
```javascript
{
  _id: ObjectId,
  user_id: ObjectId,
  suggestion_type: String,       // 类型
  content: String,               // 建议内容
  action_items: [String],        // 行动项
  is_completed: Boolean,         // 是否完成
  created_at: Date
}
```

## 四、提示词工程设计

### 4.1 基础人设Prompt
```python
BASE_PROMPT_TEMPLATE = """
你是{name}，用户的{relationship}。

你的性格特点：{personality}

你正在与一位即将毕业的大学生交流。你的使命：
1. 提供真诚的情感支持和陪伴
2. 基于对话历史，给出实用的职业发展建议
3. 帮助用户分析自身优势和发展方向
4. 鼓励用户面对挑战，建立信心

重要原则：
- 保持{personality}的性格特点
- 回复要温暖、具体、可执行
- 避免空洞的鸡汤，提供实际帮助
- 适时分享具体的例子和经验

最近的对话历史：
{conversation_history}

请以自然、亲切的语气回复用户。
"""
```

### 4.2 情感支持场景
```python
EMOTIONAL_SUPPORT_PROMPT = """
用户当前可能处于{emotion_state}状态（基于语气判断）。

请注意：
1. 先共情，理解和认可用户的感受
2. 提供情感价值，而非立即给解决方案

4. 保持{personality}的性格，但要真诚温暖
5. 适当时给予鼓励，但不要强迫积极

回复要点：
```

### 4.3 职业建议场景
```python
CAREER_ADVICE_PROMPT = """
```

## 五、核心API设计

### 5.1 认证相关
```
POST /api/auth/session
- 描述：创建会话（简化版，生成session_id）
- 请求：{}
- 响应：{ session_id, user_id }
```

### 5.2 数字人管理
```
POST /api/digital-person
- 描述：创建数字人
- 请求：{ name, relationship, personality, avatar }
- 响应：{ id, name, ... }

GET /api/digital-person/{user_id}
- 描述：获取用户的数字人列表
- 响应：[{ id, name, ... }]

PUT /api/digital-person/{id}
- 描述：更新数字人配置
- 请求：{ name, personality, custom_prompt }
```

### 5.3 对话相关
```
POST /api/conversation/message
- 描述：发送消息并获取AI回复
- 请求：{ 
    user_id, 
    digital_person_id, 
    message,
    conversation_id (可选)
  }
- 响应：{ 
    reply, 
    conversation_id,
    emotion (可选)
  }

GET /api/conversation/{id}
- 描述：获取对话历史
- 响应：{ messages: [...] }

DELETE /api/conversation/{id}
- 描述：删除对话
```

### 5.4 分析相关
```
GET /api/analysis/emotion/{user_id}
- 描述：获取情感分析报告
- 响应：{ emotion_trend, key_topics }

POST /api/analysis/career
- 描述：生成职业建议
- 请求：{ user_id, context }
- 响应：{ suggestions: [...] }
```

## 八、风险控制

### 8.1 技术风险
| 风险 | 影响 | 应对措施 |
|------|------|---------|
| AI API不稳定 | 高 | 实现降级策略，预设回复 |
| MongoDB连接失败 | 高 | 添加重试机制 |
| 语音识别准确率低 | 中 | 提供文字备选 |
| 跨域问题 | 低 | 正确配置CORS |

### 8.2 时间风险
- **风险**：开发时间不足
- **应对**：优先完成核心功能（对话系统），次要功能可后续添加

### 8.3 质量风险
- **风险**：代码质量差，难以维护
- **应对**：遵循编码规范，及时注释，定期代码review
